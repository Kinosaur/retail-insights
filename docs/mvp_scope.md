# MVP Scope — Retail Insights Engine

## Purpose of This Document

This document defines what is being built, what is not being built, and what success looks like for this project. It is the reference point for scope decisions — if a feature is not listed here, it is not part of the MVP.

---

## Problem Statement

Small clothing retailers collect sales and inventory data in spreadsheets but have no easy way to turn that data into decisions. The MVP proves that a clean backend API can automate the most common retail intelligence questions for a shop of this size.

---

## MVP Goal

A working REST API that:
1. Accepts CSV/Excel uploads for products, sales, and inventory
2. Validates and cleans the data, reporting exactly what was rejected and why
3. Answers five core retail analytics questions with real data
4. Forecasts short-term demand for any product
5. Produces a plain-English AI summary a non-technical shop owner can read

Delivered in 20 days. Demonstrated with synthetic Uniqlo Singapore 2023–2025 data.

---

## In Scope

### Data Ingestion
- Upload products, sales, and inventory via CSV or Excel file
- Row-level validation with specific error messages per rejected row
- Auto-creation of unknown products referenced in sales files
- Upsert on product upload (update existing, insert new)
- Duplicate detection and silent deduplication
- Upload audit log (every batch stored with filename, status, row counts)

### Analytics
- **Overview** — total revenue, units sold, unique products sold, average order value for any date range
- **Top products** — ranked by revenue, units sold, or profit margin
- **ABC classification** — A (top 80% revenue) / B (next 15%) / C (bottom 5%) for the full catalog
- **Dead stock** — products with inventory on hand but no sales in N days
- **Reorder alerts** — products where current stock falls below the reorder point based on recent sales velocity

### Forecasting
- Simple Moving Average (SMA) weekly demand forecast for any single product
- Configurable forecast horizon (default 4 weeks)
- Honest disclaimer that SMA is a baseline, not a production forecasting model

### AI Explanation
- Plain-English summary of the analytics output generated by Groq (llama-3.3-70b-versatile, free tier)
- Three-part structure: what is working, what needs attention, recommended action this week
- Result cached in the database (hash of metrics as cache key) — LLM is only called when the underlying data changes

### Infrastructure
- Shared Neon cloud PostgreSQL database (Singapore region, free tier)
- Alembic migration history for every schema change
- Local test database isolation — tests never touch Neon
- pytest test suite covering all upload and GET endpoints
- Interactive API documentation (FastAPI auto-generated at `/docs`)

---

## Out of Scope (MVP)

The following are explicitly not being built for MVP. They may be considered after Day 20.

| Feature | Reason Out of Scope |
|---------|---------------------|
| User authentication / login | Single-tenant demo; no multi-user requirement |
| Frontend / dashboard UI | API-only; a UI can be built on top later |
| Real supplier data for lead times | Using configurable defaults (14-day lead time, 7-day safety stock) |
| Multi-currency support | Singapore dollars only |
| Excel export / report download | Upload only; no download endpoints |
| Webhooks or real-time events | Polling-based; no push notifications |
| Multi-year comparison views | Year-over-year comparison endpoints are not built for MVP |
| Machine learning forecasting | SMA only; ML models require more infrastructure |
| Product image storage | Text data only |
| Batch upload progress tracking | Synchronous uploads only |
| Role-based access control | No access control at all in MVP |

## 20-Day Build Plan

### Days 1–2 — Foundation ✅
- Project structure and virtual environment
- Database schema design (5 tables)
- SQLAlchemy models and first Alembic migration
- API contract document agreed between Person A and Person B
- FastAPI app skeleton with `/health` endpoint

### Days 3–5 — Ingestion Pipeline ✅
- CSV and Excel parser
- Row-level validators for products, sales, and inventory
- Upload endpoints (`POST /upload/products`, `/upload/sales`, `/upload/inventory`)
- Upload batch audit log
- GET endpoints for products (list + detail) and upload batches
- 34 pytest tests covering all endpoints
- Synthetic dataset generator (Singapore Uniqlo 2023)
- Data seeded to Neon cloud

### Days 6–9 — Analytics ✅
- `GET /analytics/overview` — revenue, units, AOV for any date range
- `GET /analytics/top-products` — ranked by revenue, units, or margin
- `GET /analytics/abc` — full catalog classified into A / B / C
- `GET /analytics/dead-stock` — slow-moving inventory flagged
- `GET /analytics/reorder` — stock-level alerts based on sales velocity

### Days 10–12 — Forecast ✅
- Weekly sales aggregation per product
- Simple Moving Average calculation (8-week rolling window)
- `GET /analytics/forecast` — projected demand for a given SKU and number of weeks
- Input validation (product must exist, weeks 1–12)
- 3-year dataset regenerated (2023–2025) with correct Singapore festive calendar per year and YoY growth

### Days 13–15 — AI Layer ✅
- Groq SDK integration (free tier, llama-3.3-70b-versatile)
- Gather all analytics data (overview, top products, ABC, dead stock, reorder)
- Compute cache key as SHA-256 hash of metrics JSON
- Check `analysis_runs` table for cached result
- Call Groq API if no cache hit; validate response keys before caching
- Store result in `analysis_runs` with status "done" or "failed"
- `GET /analytics/explain` — returns structured AI summary with `cached` flag

### Days 16–20 — Polish ✅
- README rewritten: status, data stats, tech stack, limitations, what's next
- All status tables and stats updated across README, project_overview, mvp_scope
- GROQ_API_KEY added to prerequisites and .env.example
- docs/api.md "Last updated" header corrected
- Final end-to-end demo run against Neon — all 7 endpoints verified
- 34 tests passing

## Success Criteria

The MVP is considered complete when:

- [x] All 7 analytics endpoints return correct data from the live Neon database
- [x] The AI explanation endpoint produces a coherent, grounded summary (verified against live Neon data)
- [x] The upload pipeline correctly handles dirty data (rejects bad rows, accepts good ones, reports both)
- [x] All existing tests still pass after Day 20 changes (34/34)
- [x] A non-technical person can read the AI summary and know what to do next
- [x] The README is complete enough for a new developer to clone and run the project in under 10 minutes

## Known Limitations (Accepted for MVP)

These are known constraints that are acceptable for the MVP and documented honestly.

| Limitation | Impact |
|------------|--------|
| No transaction grouping | Sales rows are individual line items with no receipt/order ID — basket AOV and "frequently bought together" analysis are not possible without a schema change |
| SMA forecast is a baseline only | Does not account for seasonality, trend, or promotions — results are indicative, not predictive |
| Lead time and safety stock are defaults | Not sourced from real supplier data — shop owner should adjust via query params |
| No authentication | Anyone with the URL can call any endpoint |
| Synthetic dataset | Results reflect simulated patterns, not a real shop's actual performance |
| Single-tenant | No concept of multiple shops or users |
| No year-over-year comparison endpoint | Data spans 2023–2025 but there is no endpoint to compare performance across years |

## Data Dictionary

| Column | Table | Type | Notes |
|--------|-------|------|-------|
| `product_id` | products | String (PK) | Uppercase, e.g. `E483831-000` |
| `product_name` | products | String | Required |
| `category` | products | String (nullable) | e.g. `Women`, `Men`, `Kids` |
| `cost_price` | products | Float (nullable) | Wholesale cost |
| `sell_price` | products | Float (nullable) | Retail price |
| `sale_date` | sales | Date | Must not be in the future |
| `quantity` | sales | Integer | Negative = return |
| `unit_price` | sales | Float | Must be > 0 |
| `revenue` | sales | Float | `quantity × unit_price`, stored for query performance |
| `is_return` | sales | Boolean | True when `quantity < 0` |
| `quantity_on_hand` | inventory_snapshots | Integer | Must be >= 0 |
| `snapshot_date` | inventory_snapshots | Date | Date the count was taken |
| `status` | upload_batches | String | `pending` / `success` / `partial` / `failed` |
| `cache_key` | analysis_runs | String | Hash of the metrics JSON fed to the LLM |
